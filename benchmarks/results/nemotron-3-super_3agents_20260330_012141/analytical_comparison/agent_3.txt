### **Structured Comparison: SQLite vs. PostgreSQL vs. DuckDB (10M-row Read-Heavy Analytics)**

| Criteria               | SQLite                                                                 | PostgreSQL                                                                 | DuckDB                                                                 |
|------------------------|------------------------------------------------------------------------|----------------------------------------------------------------------------|------------------------------------------------------------------------|
| **Query Speed**        | ⚠️ Moderate to Good (single-threaded, no vectorization)<br>• Fast for simple scans/filters on indexed columns<br>• Slower on complex aggregations, joins, or window functions due to lack of parallelism and columnar storage<br>• Best when data fits in RAM and queries are simple | ✅ Good to Very Good (multi-threaded, mature query planner)<br>• Excels with proper indexing, partitioning, and materialized views<br>• Strong for complex SQL (CTEs, window functions, JSON)<br>• Can struggle with 10M-row ad-hoc analytics without tuning (e.g., missing stats, no columnar layout) | ✅✅ Excellent (optimized for OLAP)<br>• Columnar storage + vectorized execution + parallelism<br>• Built for fast aggregation, grouping, and scanning on 10M–100M rows<br>• Often 2–10x faster than PostgreSQL on analytical queries (e.g., `GROUP BY`, `SUM`, `AVG`)<br>• Near-instant for typical BI workloads on this scale |
| **Memory Usage**       | ✅ Very Low<br>• Embedded, single-file, minimal overhead<br>• Uses OS page cache; ~50–200 MB for 10M rows (depends on schema/indexes)<br>• No server process; memory scoped to app | ⚠️ Moderate to High<br>• Requires separate server process<br>• Shared buffers, work_mem, maintenance_work_mem add overhead<br>• Expect 500 MB–2 GB+ RAM for 10M rows with indexes and connection pooling<br>• Can be tuned down, but defaults are server-oriented | ✅ Low to Moderate<br>• Designed for embedded/analytics use<br>• Efficient columnar compression (often 2–5x smaller than row-based)<br>• Uses ~100–400 MB RAM for 10M rows (depends on compression and parallelism)<br>• Spills to disk gracefully if needed (out-of-core) |
| **Operational Complexity** | ✅✅ Minimal<br>• Zero-config: single `.db` file<br>• No server, no users, no permissions, no backup/restore scripts needed (just copy file)<br>• Ideal for single-app, embedded, or dev/test scenarios<br>• Limited concurrency (single writer; multiple readers OK) | ⚠️ Moderate<br>• Requires installation, configuration, startup/shutdown<br>• Needs tuning (shared_buffers, max_connections, vacuum, autovacuum)<br>• Backup via `pg_dump`, replication, monitoring (Prometheus/Grafana)<br>• Overkill for simple analytics unless already in use<br>• Strong ACID, multi-user, production-ready | ✅ Low<br>• Single binary or library (no server required)<br>• Can run embedded in Python/R/Julia/Jupyter or as CLI<br>• No config needed for basic use; optional tuning for parallelism/threads<br>• Persists as single file (like SQLite) or in-memory<br>• Backup = copy file; no vacuuming, no WAL management<br>• Excellent for notebooks, scripts, and lightweight services |
| **Ecosystem Maturity** | ✅✅ Very High<br>• Ubiquitous: embedded in Android, iOS, browsers (via WASM), Python (`sqlite3`), R, etc.<br>• Decades of battle-testing<br>• Limited to SQL: no advanced analytics extensions (no ML, no geospatial beyond basic)<br>• No native support for Parquet/JSONL/CSV streaming (though can import) | ✅✅✅ Industry Gold Standard<br>• Mature, enterprise-grade: replication, logical decoding, partitioning, extensions (PostGIS, TimescaleDB, pg_stat_statements)<br>• Rich tooling: pgAdmin, DBeaver, psql, ORMs (SQLAlchemy, Django)<br>• Cloud-native: AWS RDS, GCP Cloud SQL, Azure DB for PostgreSQL<br>• Overkill for pure analytics but unmatched for transactional + analytical hybrid workloads | ✅✅ Rapidly Growing (but not yet legacy)<br>• Explosive adoption in data science/ML pipelines (Python: `duckdb`, `polars`, `pandas` integration)<br>• Native support for Parquet, CSV, JSON, IPC (Arrow)<br>• Integrates with Ibis, Polars, Polars, Modin, SQLAlchemy<br>• Growing community, active dev (MotherDuck cloud service)<br>• Less mature than PostgreSQL for enterprise ops (logging, monitoring, RBAC) — but improving fast |

### **Summary Recommendation for 10M-Row Read-Heavy Analytics**

| Use Case                                                                 | Best Choice       | Why                                                                 |
|--------------------------------------------------------------------------|-------------------|---------------------------------------------------------------------|
| **Embedded app, script, notebook, or prototype** (e.g., Jupyter, Streamlit, CLI tool) | **DuckDB**        | Best speed/ease tradeoff; columnar + vectorized; zero config; handles 10M rows in sub-second to few seconds; integrates natively with Python/pandas. |
| **Simple queries, low concurrency, already using SQLite** (e.g., mobile app logging, config store) | **SQLite**        | If your workload is mostly point lookups or simple filters — and you value zero ops over speed — SQLite is fine. Avoid for complex aggregations. |
| **Enterprise deployment, existing PG infrastructure, need for ACLs/audit/replication** | **PostgreSQL**    | Only choose if you already have PG, need multi-user security, or require hybrid OLTP/OLAP (e.g., powering a dashboard *and* user accounts). Overkill for pure analytics. |
| **You need to query Parquet/CSV files directly without loading**          | **DuckDB**        | DuckDB excels at querying external formats natively; PostgreSQL requires `fdw` or `COPY`; SQLite has no native support. |
