### **Summary Recommendation for 10M-Row Read-Heavy Analytics**

| Use Case                            | Best Choice       | Why                                                                 |
|-------------------------------------|-------------------|---------------------------------------------------------------------|
| **Maximum query speed**             | **DuckDB**        | Columnar + vectorized + parallel = purpose-built for this workload. |
| **Lowest memory footprint**         | **SQLite**        | If data fits in RAM and queries are simple; but speed suffers on complex analytics. |
| **Simplest deployment & ops**       | **DuckDB** ≈ **SQLite** | Both are zero-config, single-file. DuckDB wins if you need analytical power. |
| **Most mature ecosystem**           | **PostgreSQL**    | If you need BI tool integration, replication, cloud managed services, or already use PG. |
| **Best overall for analytics**      | **DuckDB**        | **Sweet spot**: speed of columnar OLAP, simplicity of SQLite, growing ecosystem. Ideal for 10M-row analytical workloads. |

> 💡 **Note**: If your 10M rows are part of a larger pipeline (e.g., ETL, streaming, multi-user access, or already using PostgreSQL), then PostgreSQL remains a safe, robust choice. But for **standalone, read-heavy analytics on 10M rows**, **DuckDB is the clear winner** — it’s literally why it was built.
