### **Structured Comparison: SQLite vs. PostgreSQL vs. DuckDB for Read-Heavy Analytics (10M Rows)**

| Dimension             | SQLite                                                                 | PostgreSQL                                                                 | DuckDB                                                                 |
|-----------------------|------------------------------------------------------------------------|----------------------------------------------------------------------------|------------------------------------------------------------------------|
| **Query Speed**       | ⚠️ **Moderate to Slow**<br>- Single-threaded, no vectorized execution<br>- Indexes help for point lookups but not ideal for aggregations/scans<br>- 10M rows: ~1–5 sec for simple aggregations on SSD; degrades with complex joins/window functions | ✅ **Good to Very Good**<br>- Mature query planner, parallel query (since PG 9.6)<br>- Bitmap scans, hash aggregates, and parallel execution scale well<br>- 10M rows: ~0.2–1.5 sec for typical OLAP queries with proper indexes/partitions | ✅✅ **Excellent**<br>- Columnar storage + vectorized execution + multi-threaded<br>- Optimized for analytics: pushdown predicates, efficient joins, no index needed for scans<br>- 10M rows: often **< 0.1–0.5 sec** for aggregations, even complex ones |
| **Memory Usage**      | ✅ **Very Low**<br>- Embedded, minimal overhead (~1–10 MB baseline)<br>- Entire DB can fit in RAM easily for 10M rows (if narrow schema)<br>- No server process; memory used only by client connection | ⚠️ **Moderate to High**<br>- Shared buffers, work_mem, maintenance_work_mem, etc.<br>- Minimum ~50–100 MB for small instance; scales with concurrency and workload<br>- For 10M rows: ~100–500 MB typical for read-heavy analytics (depends on config) | ✅ **Low to Moderate**<br>- Designed for in-memory analytics; spills to disk only if exceeds RAM<br>- Baseline ~10–50 MB; efficient columnar compression reduces footprint<br>- 10M rows: often fits in < 100 MB RAM even with wide tables |
| **Operational Complexity** | ✅✅ **Very Low**<br>- Single file; no server, no config, no users/permissions to manage<br>- Copy file = backup; ideal for embedded or single-user analytics<br>- Zero setup; ideal for scripts, notebooks, or CLI tools | ⚠️ **Moderate**<br>- Requires running a server (pg_ctl, systemd, Docker)<br>- Needs tuning (shared_buffers, max_parallel_workers, etc.)<br>- Backup/restore (pg_dump, WAL), vacuuming, monitoring add overhead<br>- Overkill for simple 10M-row read-heavy use unless multi-user or ACID writes needed | ✅ **Low**<br>- Embedded library (like SQLite) or standalone CLI<br>- No server required; single binary or Python/R/Julia package<br>- Minimal config; automatic memory management<br>- Backup: just copy the `.db` file (if using persistent storage) |
| **Ecosystem Maturity** | ✅✅ **High**<br>- Ubiquitous: built into Android, iOS, browsers, Python stdlib<br>- ODBC/JDBC drivers, DB-API, SQLAlchemy, pandas `read_sql`<br>- Limited analytics-specific tools (no native columnar stats, no parallel query) | ✅✅✅ **Very High**<br>- 30+ years of development; exhaustive tooling (pgAdmin, psql, pgBouncer, logical replication)<br>- Rich extensions (PostGIS, TimescaleDB, pg_partman)<br>- ANSI SQL compliant; strong JSON, GIS, full-text search<br>- Enterprise support, cloud vendors (AWS RDS, GCP Cloud SQL) | ⚠️ **Moderate → Growing Fast**<br>- Rapid adoption in data science/ML (Python/R/Julia integration)<br>- Excellent pandas/Polars/Arrow interoperability<br>- Native support for Parquet, CSV, JSON, Iceberg<br>- Fewer GUI tools; no enterprise support yet (though MotherDuck offers managed service)<br>- Community very active; docs and examples excellent |

---

### **Summary Recommendation for 10M-Row Read-Heavy Analytics**

| Use Case                            | Best Choice      | Why                                                                 |
|-------------------------------------|------------------|---------------------------------------------------------------------|
| **Maximum query speed**             | **DuckDB**       | Columnar + vectorized execution outperforms others on analytical scans/aggregations |
| **Lowest memory footprint**         | **SQLite**       | Minimal overhead; ideal if data fits in RAM and no parallelism needed |
| **Simplest setup & maintenance**    | **SQLite** or **DuckDB** | Both are serverless; DuckDB adds analytics optimizations without complexity |
| **Most mature & enterprise-ready**  | **PostgreSQL**   | Best for multi-user, long-term, mission-critical systems with complex needs |
| **Best overall for analytics**      | **DuckDB**       | Optimal balance: speed, low ops, good memory efficiency, and strong analytics focus — purpose-built for this workload |

> 💡 **Note**: If your 10M rows are static or append-only (common in analytics), **DuckDB** is often the superior choice. Use **PostgreSQL** only if you need concurrent writes, complex transactions, or integration with existing OLTP systems. Use **SQLite** only if the workload is extremely simple (e.g., filtered lookups) and you prioritize zero setup over query performance.
