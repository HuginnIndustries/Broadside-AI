### ✅ **Function Under Review**

def get_users(db, filters={}):
    query = f'SELECT * FROM users WHERE 1=1'
    for key, val in filters.items():
        query += f" AND {key} = '{val}'"
    results = db.execute(query)
    users = []
    for row in results:

## 🔴 **Critical Severity Issues**

### 1. **SQL Injection Vulnerability**
- **Issue**: The function directly interpolates user-controlled `filters` dictionary values into the SQL query string using f-strings without any sanitization or parameterization.
- **Impact**: An attacker can inject arbitrary SQL via `filters` keys or values (e.g., `filters={"username": "admin' OR '1'='1"}`) to bypass authentication, extract data, modify/delete records, or execute arbitrary commands.
- **Fix**: Use **parameterized queries** (prepared statements) via the database driver’s API (e.g., `db.execute(query, params)`).
- **Severity**: **Critical**

### 2. **Mutable Default Argument (`filters={}`)**
- **Issue**: The default value `filters={}` is a mutable dictionary. In Python, default arguments are evaluated **once** at function definition time, not at each call.
- **Impact**: If the function modifies `filters` (which it doesn’t explicitly, but callers might), or if the same default dict is reused across calls, it leads to **state persistence** and **unintended side effects**. For example:
  ```python
  get_users(db)  # first call
  get_users(db)  # second call — filters may retain state from first call if modified elsewhere
  ```
- **Fix**: Use `filters=None` and initialize inside the function: `if filters is None: filters = {}`.
- **Severity**: **Critical** (due to potential for subtle, hard-to-debug bugs in production)

## 🟠 **High Severity Issues**

### 3. **Plaintext Password Exposure in Memory**
- **Issue**: The function decrypts the password and stores it in plaintext in the `user` dictionary (`user['password'] = decrypt(user['password'])`), then returns the entire user object.
- **Impact**: If the returned `users` list is logged, serialized, or passed to an insecure component (e.g., frontend, API response), **plaintext passwords are exposed**. Even if not returned, keeping decrypted passwords in memory longer than necessary increases attack surface (e.g., via memory dumping).
- **Fix**: 
  - Do **not** return decrypted passwords unless absolutely necessary.
  - If decryption is needed for internal use (e.g., comparison), decrypt only in a secure, isolated scope and **never store or return** the plaintext.
  - Consider whether decryption is needed at all — often, you only need to compare hashes (not decrypt).
- **Severity**: **High**

### 4. **No Input Validation or Sanitization on Column Names**
- **Issue**: The `key` from `filters.items()` is directly interpolated into the SQL query as a column name (`{key}`). While SQL injection via values is the primary concern, **column names are also dangerous** if not validated.
- **Impact**: An attacker could inject malicious column names (e.g., `filters={"username; DROP TABLE users;--": "value"}`) leading to **schema manipulation or data destruction**.
- **Fix**: Maintain a **whitelist of allowed column names** (e.g., `ALLOWED_COLUMNS = {'id', 'username', 'email', 'created_at'}`) and reject any `key` not in the whitelist before using it in the query.
- **Severity**: **High**

### 5. **No Error Handling**
- **Issue**: The function assumes `db.execute(query)` always succeeds and that `decrypt()` never fails.
- **Impact**: 
  - If the database is down, query syntax is invalid, or connection times out → unhandled exception crashes the function/caller.
  - If `decrypt()` raises an exception (e.g., invalid padding, wrong key), the entire function fails silently or crashes, potentially leaking stack traces.
- **Fix**: Wrap database and decryption calls in `try/except` blocks. Log errors appropriately and return a meaningful error (e.g., empty list, or raise a custom exception).
- **Severity**: **High**

## 🟡 **Medium Severity Issues**

### 6. **Inefficient String Concatenation in Loop**
- **Issue**: Using `query += f" AND {key} = '{val}'"` in a loop creates a new string object on each iteration due to Python’s string immutability.
- **Impact**: For large `filters` dictionaries (e.g., 100+ keys), this leads to **O(n²)** time complexity and unnecessary memory allocation.
- **Fix**: Build a list of condition strings and join them once:
  ```python
  conditions = []
  params = []
  for key, val in filters.items():
      if key in ALLOWED_COLUMNS:  # after whitelist validation
          conditions.append(f"{key} = ?")
          params.append(val)
  query = "SELECT * FROM users WHERE 1=1" + (" AND " + " AND ".join(conditions) if conditions else "")
  results = db.execute(query, params)
  ```
- **Severity**: **Medium** (only noticeable with large filters; still worth fixing for scalability)

### 7. **Selecting All Columns (`SELECT *`)**
- **Issue**: The query uses `SELECT * FROM users`, which retrieves all columns — including potentially sensitive or large ones (e.g., `profile_pic`, `bio`, `ssn`) even if not needed.
- **Impact**: 
  - Wastes network I/O and memory.
  - Increases risk of accidental data exposure if the result is logged or returned to client.
  - Makes the function brittle: if the table schema changes (e.g., adding a large BLOB column), performance degrades silently.
- **Fix**: Explicitly list only the columns needed (e.g., `SELECT id, username, email, created_at FROM users`).
- **Severity**: **Medium**
