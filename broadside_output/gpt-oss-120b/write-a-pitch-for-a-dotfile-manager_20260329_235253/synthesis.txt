## **DotForge / DotSync / DotKeeper – a unified, cross‑platform dotfile management platform**

### The Vision  
Give every developer a **single, declarative source of truth** for all configuration files, secrets and required packages, so a fresh machine becomes productive with one command—without manual copying, merge conflicts, or secret leakage.

### Core Experience  

| Step | Command | What happens |
|------|---------|---------------|
| **Install** | `curl -fsSL https://dotfiles.dev/install | sh` (Linux/macOS/WSL) | Installs the CLI (`dotx`) and, if the user wants, the optional Electron/TUI dashboard. |
| **Bootstrap** | `dotx init <repo‑url>` | Detects OS, clones the repo, creates a **declarative manifest** (`dotx.yml`) if it doesn’t exist, and generates an encrypted **secrets store** (AES‑256, age, or SOPS – user‑selectable). |
| **Sync / Apply** | `dotx apply` (or `dotx apply --ci` in CI pipelines) | 1️⃣ Installs OS‑specific packages (brew, apt, pacman, choco, scoop) as declared in the manifest. 2️⃣ Renders all templated files (`{{ os }}`, `{{ hostname }}`, `{{ env.VAR }}`). 3️⃣ Links files/symlinks atomically. 4️⃣ Stores a signed snapshot in the underlying Git repo. |
| **Rollback** | `dotx rollback <snapshot-id>` | Restores the entire environment (files, links, packages, secrets) to the chosen snapshot instantly. |
| **Inspect** | `dotx status / dotx diff / dotx log` *(or via the UI)* | Shows what would change, history of commits, and secret‑vault contents (read‑only). |
| **Extend** | `dotx plugin add <name>` | Pulls a plugin from the public marketplace (e.g., `brew-sync`, `npm-link`, `kube-config-manager`) or a custom script written in Bash, Python, Rust, etc. |

### Feature Highlights (combined best‑of‑all)

| Feature | Why it matters | Implementation (any of the three drafts) |
|---------|----------------|------------------------------------------|
| **Declarative Manifest (YAML/JSON)** | Guarantees reproducibility; a single file versioned in Git. | DotForge’s explicit `dotforge.yml`. |
| **Cross‑platform templating** | One source can generate Linux, macOS, WSL, Windows configs. | Jinja‑style blocks used by all three. |
| **Built‑in package‑manager integration** | Ensures required binaries, fonts, tools are installed before configs are linked. | Core in DotForge; plug‑in approach in DotSync. |
| **Zero‑trust encrypted secret vault** | Secrets never appear in clear‑text on the repo or filesystem. | AES‑256 (DotSync), age/PGP (DotKeeper), SOPS (DotForge). |
| **Atomic snapshots & signed roll‑backs** | One‑click safety net for any change. | Snapshots (DotSync), signed commits (DotKeeper), Git‑based atomic apply (DotForge). |
| **CLI‑first workflow with optional UI** | Works for power users and newcomers alike. | Electron dashboard (DotSync), curses TUI (DotKeeper), optional Electron UI (DotForge). |
| **Plugin / Marketplace ecosystem** | Extensible to any toolchain; community‑driven growth. | All three mention a marketplace; concrete plug‑ins listed in DotSync. |
| **Team & Enterprise features** | Shared baselines, per‑member overrides, audit logs, SSO, self‑hosted control plane. | DotSync Server, DotKeeper Enterprise SSO, DotForge SaaS/On‑prem. |
| **Free tier + Pro/Enterprise** | Low barrier to entry, revenue path. | $9 / month per user for Pro (DotSync & DotKeeper); similar pricing implied for DotForge. |
| **CI/CD friendly** | Validate that production boxes match the repo. | `dotx apply --ci` (DotKeeper), scriptable CLI (all). |
| **Open‑source core** | Trust and community contributions. | MIT‑licensed core in DotKeeper & DotForge; implied in DotSync. |
